# Правила интеграции и работы с функциями-заглушками 

Так как демонстрация рабочих версий продукта и интеграция проводятся каждую пару, команда придерживается строгих правил взаимодействия.

## 1. Процесс интеграции на занятиях
- **Подготовка:** Перед занятием каждый разработчик обязан убедиться, что его ветка `feature/...` работает и проходит локальные тесты.
- **Слияние:** На паре или в общаге происходит совместная проверка кода (Code Review) и слияние готовых веток в общую ветку `develop`.
- **Проверка:** Тестировщик проверяет интегрированный функционал по заранее написанным black-box тестам из папки `tests/black_box/`.
- **Релиз:** Стабильная версия из `develop` переносится в `main` для демонстрации преподавателю.

## 2. Форматы взаимодействия (Контракты)
Если задача `А` (например, интерфейс) зависит от задачи `Б` (например, логика базы данных), которая еще не выполнена, разработчики фиксируют **контракт** в Wiki:
1. Название метода/функции.
2. Входные параметры (типы данных).
3. Возвращаемое значение (формат).

## 3. Функции-заглушки (Stubs)
До этапа полной интеграции разработчик зависимой задачи обязан использовать функции-заглушки. Это временные функции, которые имитируют работу реального алгоритма, возвращая заранее заготовленные (захардкоденные) данные.

**Пример оформления заглушки (Python):**
```python
def get_user_habits_mock(user_id: int) -> list[dict]:
    """
    ЗАГЛУШКА: Имитирует получение списка привычек пользователя из базы данных.
    Будет заменена на реальный запрос к БД после завершения задачи по архитектуре.
    """
    return [
        {"habit_id": 1, "name": "Пить воду", "streak": 5},
        {"habit_id": 2, "name": "Читать 20 страниц", "streak": 2}
    ]
```
